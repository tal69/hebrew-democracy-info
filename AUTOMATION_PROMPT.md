# Daily Democracy Paper Additions Automation Prompt

Use this prompt when recreating the nightly Codex automation in another OpenAI account.

Recommended schedule: daily at 05:00 Asia/Jerusalem.

Recommended working directory: the top-level `DemocracyWebSite` folder if available, with the prompt instructing Codex to work in `github_pages_publish`. If cloning only the GitHub repo, set the working directory to the repository root and remove the `Work in github_pages_publish` instruction.

```text
Find 3 new academic papers by Israeli researchers with a liberal perspective on democracy, liberal democracy, rights, institutions, constitutionalism, rule of law, democratic backsliding, equality, minority rights, courts, civil society, or related subjects. Avoid duplicating papers already present on the website. Prefer papers with stable public URLs, DOI links, journal/publisher pages, SSRN, institutional repository pages, or official researcher pages.

Work in `github_pages_publish`. The website source is Jekyll content, not generated HTML. Start by reading only the compact context needed for duplicate checks and conventions: `README.md`, `_data/paper_index.json`, `_data/topics.json`, `_data/site.json`, and `image_catalog.json`. Do not scan every `_papers/*.md` file unless the validator reports an error, a duplicate check is ambiguous, or a specific existing record needs inspection.

Add each new summary as a Markdown file under `github_pages_publish/_papers/` with JSON front matter between `---` markers. Use the existing paper files only as schema examples when needed. Include at least: `layout: "paper"`, `title`, `titleHe`, `descriptionHe`, `summaryHe`, `authorsCardHe`, `authorsHtml`, `paperTitle`, `journal`, `dateText`, `doiUrl` when available, `authors`, `sourceAuthors`, `topics`, `keywords`, `image`, `dateModified`, `lastUpdatedHe`, `oneLinerHtml`, `sections`, `slug`, `file`, `permalink`, `paper_url`, and a numeric `sortKey`.

Use `_data/paper_index.json` as the primary duplicate and ordering index. New papers must receive larger `sortKey` values than existing papers, such as `YYYYMMDD0001`, `YYYYMMDD0002`, and `YYYYMMDD0003`, ordered so the newest three appear first. Do not rewrite older `_papers/*.md` files merely to renumber or reorder them. After adding or editing paper files, run `python3 scripts/validate_sources.py --write-index` to regenerate `_data/paper_index.json`, then run `python3 scripts/validate_sources.py` to verify that the index is current and source data is consistent.

Assign every new paper to one or more existing topic IDs from `_data/topics.json` when substantively appropriate. A paper may appear under more than one topic. Topic page membership is derived from each paper's `topics` list; do not maintain duplicate per-topic paper lists. Add or edit `_data/topics.json` only when a new durable central topic is genuinely needed, and then also create a matching topic graphic and topic page Markdown file under `topics/` following the existing format.

Before generating a new paper graphic, consult `image_catalog.json`; reuse an existing image when the topic fit is strong and the image is not used by any of the newest 6 home-page cards listed in `_data/paper_index.json`. If no suitable reusable image exists, create a new JPEG in `html_qa/` and add it to `image_catalog.json`. Paper graphics must be 800x600 landscape JPEGs and match the site's visual language: polished high-end editorial illustration rather than crude cartoons, subtle watercolor/gouache texture, warm cream and ochre tones, muted teal/deep-blue accents, Mediterranean civic architecture, dignified human figures when relevant, natural light, thoughtful symbolic composition, no text, no letters, no logos, no flags, no watermarks, no childish clip-art, no crude flat vector shapes, no heavy outlines, no ghosted duplicate artwork, no translucent overlays, no UI widgets, and no pasted-looking layers.

Apply the GEO/AI-crawler writing guide when creating paper data: use descriptive search-friendly Hebrew titles and headings; put a clear 2-3 sentence conclusion near the top that starts from the finding; include explicit visible credit with authors, institutional affiliation when confidently verified, journal, year/date, and active DOI; repeat full author names and key institutions/concepts enough that quoted passages remain attributable; make each paragraph's first sentence a standalone thesis sentence; keep paragraphs short; avoid using images for data/tables; keep DOI/source links active; and include FAQ-style question headings/answers when the format allows. For Hebrew pages, include English author names in the metadata and link Hebrew/English identity cues where useful.

On each new article record, link author names in the article-page metadata row to official academic home pages only when you are certain about the researcher's identity. Prefer official institutional profiles or clearly maintained academic personal home pages; leave a name unlinked when uncertain. Do not add author links on the home page cards. When adding author-page links, also add the same URL to the corresponding JSON-LD author object in the paper data. Make every external HTML link in stored HTML fields open in a new tab by adding `target="_blank"` and `rel="noopener noreferrer"`; this includes DOI/paper links, author home pages, and institutional links where relevant.

The shared Jekyll layouts already add the CARRD/Tel Aviv University logo/header, back-home link, topic links, accessibility link, Open Graph tags, JSON-LD, bottom last-updated line, the full Hebrew disclaimer, and the site-wide GoatCounter analytics snippet from `_includes/analytics.html`. Do not add analytics snippets to individual paper records or duplicate them in generated content. If a site-wide format change is needed, edit `_layouts/`, `_includes/`, `_data/site.json`, or `assets/css/site.css`; do not edit every paper page manually.

Validate before committing. At minimum, run `python3 scripts/validate_sources.py --write-index` after source changes and then `python3 scripts/validate_sources.py`. If local Jekyll is available, run `bundle exec jekyll build` and then `npm_config_cache="$PWD/.npm-cache" npx -y pagefind --site _site --output-subdir pagefind`, then remove `.npm-cache`; do not commit `_site/` or `pagefind/`. If local Jekyll is unavailable because of the machine Ruby environment, note that and rely on the GitHub Pages workflow after source validation.

Commit the source changes with a clear message and push to the current branch. The GitHub Actions Pages workflow will build Jekyll, regenerate Pagefind inside `_site`, and deploy GitHub Pages. After pushing, monitor the Pages workflow when possible and report whether it succeeded. If fewer than 3 suitable non-duplicate papers can be found, add only the suitable papers and clearly note why fewer were added in the commit message or final run summary.
```
